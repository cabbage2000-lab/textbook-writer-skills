# ADR 0005:.progress.json 状态机,主 skill 是唯一写者

## 状态

已接受。

## 背景

中断可续是硬需求(PRD 用户故事 US6):10+ 章教材不可能一个会话写完,中断随时发生,作者期望重新触发后从断点无缝续写。

进度状态放哪里、谁来写,有两个被否决的方向:靠**对话记忆**——新会话即丢失,无从谈续写;**多方共写状态文件**——子 skill 既可被主 skill 调度、也可独立使用,两种模式并存时多写者必然产生冲突写入和状态口径分裂。

## 决策

- `.progress.json` **只由主 skill write-textbook 读写**,三个子 skill 一律不碰;独立使用子 skill 时不产生此文件;
- 写盘时机纪律:每完成一个阶段、每完成一章,**立即**更新写盘,绝不批量延迟——中断可能发生在任何时刻;
- 重入按[契约文档第 4 节](../../skills/write-textbook/references/handoff-contract.md)的规则表定位续点:gate 未确认的阶段重新呈现产出等确认,已完成章不重写,`chapters.next` 指向最小未完成章号;
- gate 重入时的修改-再确认循环始终委托 design-textbook-outline 执行,确认后由主 skill 更新状态——内容职责与状态职责不混。

## 后果

状态逻辑集中在主 skill 一处,子 skill 保持无状态、可独立复用;调试"续写不对"类问题只需核对一个 JSON 文件与规则表。

`.progress.json` 的 schema 是对外落盘格式的一部分:不兼容变更即 MAJOR 版本(存量教材项目无法续用)。重入时 gate 未确认必须重新走确认流程——宁可让作者多确认一次,也不把未确认的产物当已确认用,这是中断可续与 gate 纪律([ADR 0002](0002-ubd-backward-design-with-dual-gates.md))的交点,不得为"续写流畅"而放松。
