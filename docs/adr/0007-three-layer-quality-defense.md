# ADR 0007:三层质量防线,行为测试保持人工运行

## 状态

已接受。

## 背景

本仓库的产品是 `skills/` 下的 Markdown skill——"给模型看的程序",没有传统应用代码。改一行指令就可能改变 skill 的运行行为,但这种回归无法用静态手段发现:结构校验和单元测试只能保证**文件形态**正确(frontmatter 合规、引用不断、无占位符),管不了 gate 停不停、契约断不断、答案真不真这些**行为质量**。

行为质量只能真实运行检验,而真实运行需要新开 Claude Code 会话、人工对照断言判定产物——放不进 CI。

## 决策

三层防线,前两层 CI 自动、第三层人工:

1. **结构校验** `scripts/validate_skills.py`:skills 内容(frontmatter、SKILL.md 500 行预算、引用完整性、占位符)+ plugin 清单一致性 + 仓库健康。零第三方依赖,Python 3.10+ 标准库,避免质量工具自身引入维护负担;
2. **单元测试** `tests/`:测校验逻辑本身,保证第一层防线不失效;
3. **行为 evals** `evals/evals.json`:M2–M5 验收场景固化为 5 个带客观断言的用例。在新开会话中输入用例 `prompt` 原文真实运行,对照 `assertions` 逐条客观判定(通过/不通过 + 产物证据),任何一条不通过即用例不通过,**不许改断言迁就产出**;用例定义入库,运行产物(`evals/workspace/`)不入库。

evals 的人工成本靠 [CONTRIBUTING.md](../../CONTRIBUTING.md) 的影响半径映射表控制:按改动位置(契约 > 调用契约段 > 流程段/references > scripts/docs)确定必跑用例,而不是每次全量重跑。

## 后果

CI 绿不等于行为正确——修改 skill 的 DoD 必须包含按映射表重跑受影响的 evals,这比纯自动化流程多一道人工环节,是本仓库接受的常态成本。

"不许改断言迁就产出"是第三层的生命线:evals 的意义就在于把"行为对不对"锚定在改动**之前**定义的客观断言上,断言可以随需求演进而修订,但必须作为独立决策记录,不能在跑不过的当下顺手改掉。
