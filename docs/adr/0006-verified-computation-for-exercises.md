# ADR 0006:例题答案必须真算验证,验证状态二值化

## 状态

已接受。

## 背景

LLM 生成的"解答"可能步骤通顺、答案错误。教材例题的答案错误是信任的致命伤:读者照着示范例题复算发现对不上,受损的不是一道题,而是整本教材的可信度(PRD 用户故事 US4:"答案是真算过的,不是编的")。

"生成即输出、事后抽查"挡不住这一点——抽查有漏网,且模型自述"已验证"并不可信。tutorial-writer 已验证过的真实性红线(操作步骤要么真跑、要么标注未验证)在 STEM 教材场景需要强化为对每道题的硬性纪律。

## 决策

- 每道计算类题目在输出前必须**真算核对**:逐步推导并复核最终答案,复算代码随题附上,供读者与作者复验;算不对的题不能输出;
- 验证状态**二值化,无第三种**:`✅ 已验证(<方式>)` 或 `⚠️ 需作者确认(<原因>)`(定义在[交接契约](../../skills/textbook/references/handoff-contract.md)中,是例题 skill 输出契约的一部分);
- 开放讨论、需要数值实验等真算不了的内容,明确标注 `⚠️ 需作者确认`,绝不假装已验证;
- 独立习题至少附参考答案或关键解题路径,不出现"留给读者"的悬空;
- 阶段 5 审核复查全书无未验证残留。

## 后果

出题成本显著高于纯生成,这是有意付出的代价;换来的是每个答案都可复算、可追溯验证方式。

这个纪律直接决定了 v1 的学科边界:锁定 STEM 概念型学科,正是因为其例题"有标准解、可验证";文科论述题、案例分析题无法真算,留给 v2 设计适配的验证方式(而不是放宽二值化纪律来容纳它们)。任何为提速而弱化验证、或引入"大概率正确"之类第三种状态的提案,都触碰 PRD 红线。
