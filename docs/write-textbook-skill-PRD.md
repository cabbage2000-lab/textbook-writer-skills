## 0. 关键决策记录（已拍板）

本 PRD 在调研报告标注的"待确认"项上做了如下决策，后续设计与验收均以此为准：

| 决策点 | 选择 | 说明 |
|--------|------|------|
| **学科范围（v1）** | 先锁 STEM 概念型 | 覆盖数学/物理/计算机等"例题有标准解、习题可验证"的学科。文科（历史/哲学等）的论述题、案例分析题留到 v2 泛化。 |
| **篇幅策略（v1）** | 拆成 skill 组合 | 不走单 skill 路线。拆成「大纲设计 + 单章写作 + 例题生成」三个子 skill，由一个主 skill 调度。**注：此决策比调研报告 v1 倾向（单 skill + 增量更新）更激进**，理由见 §4.1。 |
| **PRD 形态** | 给自己、可执行级 | 含背景/目标/用户故事/功能/非功能/里程碑/风险，可据此落地。 |

> 与调研报告的差异说明：调研报告第三部分倾向"v1 先做单 skill + 增量更新，跑通再拆分"。本 PRD 改为 v1 直接拆分，原因是——拆分能从根上规避单 skill 在长教材（10+ 章）上的上下文爆炸，且单章写作 skill 可独立复用（不只服务于本教材项目）。代价是 v1 工程复杂度更高，已在 §8 风险中标注。

## 1. 背景与问题

### 1.1 要解决的问题

写教材这件事，**难点不在写作本身，在教学设计**。现有开源 skill 解决了"把素材组织成长文档"的工程问题，但几乎没有人把"怎么把一门知识教明白"的教学法内核真正装进 skill 里。用现有工具直接生成的"教材"，普遍存在三个毛病：

1. **知识点堆砌，无主线**——一章章罗列概念，没有一条贯穿全书的主线。
2. **例题习题无梯度**——全堆在某一认知层级（通常是"理解"层），没有从低到高的认知阶梯。
3. **评估与内容脱节**——先写内容再凑习题，章节内容和"学生该学会什么"对不上。

### 1.2 差异化定位

一句话：**一个以 UbD 逆向设计为内核、面向 STEM 概念型教材的 skill 组合——先逼作者想清楚"学生该带走什么"，再倒推章节、例题、习题，产出一门有主线、有梯度、可评估的成体系教材。**

不和其他 skill 拼"长文生成"，拼"教学设计驱动"。

### 1.3 差异化来源（调研结论）

| 别人做好的（直接复用） | 没人做好的（本 skill 的空间） |
|----------------------|------------------------------|
| tutorial-writer 的工程结构：五阶段流水线、强制 gate、交接契约、真实性红线 | UbD 教学法内核：大概念 / 持久理解 / 核心问题 / 迁移目标 |
| mcpmarket course-generator 的模块化内容块 | Bloom 认知层级作为学习目标和例题梯度的标尺 |
| Codecademy lesson plan 的"学习目标 + 练习 + 测验"模块 | 脚手架 / 认知负荷驱动的章节内部节奏 |

## 2. 目标与非目标

### 2.1 目标（v1）

- **G1｜教学法驱动**：skill 必须强制作者先完成 UbD 阶段一（预期结果设计）并经人工确认后，才能进入章节编写。这是核心 gate，不可绕过。
- **G2｜有认知梯度**：全书每章的例题/习题都带 Bloom 层级标注，且系统自动检查"全书例题是否覆盖记忆→理解→应用→分析多个层级"，对全堆某一层的情况告警。
- **G3｜章节内部固定节奏**：每章正文遵循"概念讲解 → 示范例题（完整解题）→ 引导练习 → 独立习题"四段式，固化进模板。
- **G4｜可验证的真实性**：STEM 例题的答案必须真算一遍验证正确，绝不编造；习题附参考答案或解题路径。
- **G5｜长教材不爆上下文**：通过 skill 组合（大纲 + 单章 + 例题）+ 章节独立写入文件，支持 10+ 章长教材。
- **G6｜工程可靠**：复用 tutorial-writer 已验证的交接契约、术语表、配图双轨等工程机制。

### 2.2 非目标（v1 明确不做）

- ❌ 不覆盖文科教材（历史/哲学/文学的论述题、案例分析题）——v2 再做。
- ❌ 不做"素材 → 课程"的整理工具（那是 mcpmarket course-generator 的定位）——本 skill 假设作者已具备学科知识，需要的是"把知识教明白"的结构化流程。
- ❌ 不做交互式在线课程平台、不做 LMS 集成——产出物是 Markdown 教材文件。
- ❌ 不自动生成插图（截图走人工标注块、抽象流程走 SVG 示意，均需作者提供或确认）——和 tutorial-writer 的配图双轨一致。

## 3. 用户与使用场景

### 3.1 目标用户

**自己**（v1 唯一用户）：有一定学科知识、想用 AI 辅助把一门 STEM 知识写成体系化教材的人。可能是：

- 写技术教程/讲义的开发者
- 整理课程笔记成教材的学生/教师
- 做科普/教学内容的创作者

### 3.2 核心用户故事

**US1｜主线先行（最高优先级）**
> 作为作者，我希望 skill 在一开始就逼我回答"学生学完这本书该带走什么"（大概念、持久理解、核心问题、迁移目标），并要我确认后才能往下走——这样我的教材才不会变成无主线堆砌。

**US2｜梯度可见**
> 作为作者，我希望 skill 在规划章节时就给每章的例题/习题标好 Bloom 层级，并在全书层面告诉我"你的例题全在理解层、缺应用层"，让我能提前调整。

**US3｜单章可独立写**
> 作为作者，我希望写 10+ 章教材时，能一章一章地写、每章独立成文、互不污染上下文，而不是一次性塞进一个超长对话。

**US4｜例题可信**
> 作为作者，我希望 skill 生成的 STEM 例题答案是真算过的，不是编的——读者照着做不会发现答案错了。

**US5｜节奏一致**
> 作为作者，我希望每章读起来有同样的内在节奏（先讲概念，再给完整解题示范，再让我练手，再独立做），而不是有的章全是概念、有的章全是题。

**US6｜中断可续**
> 作为作者，我希望写到第 5 章中断后，下次能从大纲文件和已完成章节文件无缝接续，不用从头重来。

## 4. 总体设计：Skill 组合架构

### 4.1 为什么拆成 skill 组合（而非单 skill）

调研报告 v1 倾向"单 skill + 增量更新"。本 PRD 选择拆分，三条理由：

1. **上下文隔离**：单章写作只需该章的大纲切片 + UbD 五件套，不需要全书正文。拆分后单章写作 skill 的输入可压到最小，长教材不再爆上下文。
2. **单章 skill 可独立复用**：将来写"一篇深度技术文章"也能直接用单章写作 skill，不绑定教材项目。
3. **例题生成可专项强化**：STEM 例题的"真算验证"是独立的高难度子任务，单独成 skill 后可以专门优化（调用计算、逐步验证、附解题路径）。

代价：需要一个主 skill 做调度，状态管理（已完成哪些章）更复杂。接受。

### 4.2 四个 skill 的职责划分

```
write-textbook (主 skill / orchestrator)
  ├─ 职责：驱动五阶段流程、管理状态、在阶段间传递交接契约
  └─ 调用下面三个子 skill

  ├─ design-textbook-outline (子 skill 1：大纲设计)
  │    └─ 阶段 1-3：教学定位 → UbD 预期结果(gate) → 章节树+例题/习题计划(gate)
  │
  ├─ write-textbook-chapter (子 skill 2：单章写作)
  │    └─ 阶段 4：输入单章大纲切片 + UbD 五件套，输出该章正文
  │
  └─ generate-textbook-exercises (子 skill 3：例题生成)
       └─ 被单章写作 skill 调用：生成并验证该章的示范例题/引导练习/独立习题
```

### 4.3 状态与文件布局

教材项目落盘为一个目录，主 skill 靠文件状态做"中断可续"：

```
<教材名>/
  ├─ 00-教材设计.md          # 阶段 1-3 产出：定位 + UbD 五件套 + 章节树 + 例题/习题计划
  ├─ 01-第一章.md            # 阶段 4 产出，逐章生成
  ├─ 02-第二章.md
  ├─ ...
  ├─ 术语表.md                # 全程维护
  └─ .progress.json           # 主 skill 维护：已完成阶段、已完成章节、下次从哪续
```

## 5. 详细功能需求

### 5.1 五阶段流程（主 skill 驱动）

| 阶段 | 名称 | 执行 skill | 对应 UbD | 产出 | Gate? |
|------|------|-----------|---------|------|-------|
| 1 | 教学定位确认 | design-textbook-outline | 阶段一前奏 | 学科 / 读者认知起点 / 教材深度 / 篇幅规模 | 否（一轮提问） |
| 2 | **预期结果设计** | design-textbook-outline | **UbD 阶段一** | 大概念 + 持久理解 + 核心问题 + 迁移目标 + 学习目标（带 Bloom 动词） | **是（核心 gate）** |
| 3 | 评估与章节设计 | design-textbook-outline | UbD 阶段二+三前奏 | 章节树 + 每章例题/习题计划（带 Bloom 梯度）+ 表现性任务 | 是（次要 gate） |
| 4 | 章节正文编写 | write-textbook-chapter（逐章）| UbD 阶段三 | 各章正文 | 否 |
| 5 | 审核定稿 | write-textbook（主 skill）| — | 通读走查 + Bloom 梯度复核 + 术语一致性 + 交付 | 否 |

### 5.2 阶段 2 的 UbD 五件套（核心 gate，必须人工确认）

阶段 2 产出固定五件套，缺一不可，且必须由作者确认后才放行：

1. **大概念（Big Ideas）**：能贯穿全书、有解释力的 3–7 个核心概念。
2. **持久理解（Enduring Understandings）**：学生学完仍能保留的核心洞见，用"学生将理解……"句式。
3. **核心问题（Essential Questions）**：指向大概念的开放性问题，2–5 个，非有标准答案的题。
4. **迁移目标（Transfer Goals）**：学生能把所学用到什么新情境。
5. **学习目标（Learning Objectives）**：每条用 Bloom 对应层级的动词（"能解释/能应用/能比较"而非"理解"），可验证。

**Gate 规则**：阶段 2 产出后，主 skill 必须停下，把五件套完整呈现给作者，等明确确认（如"确认/通过"）才进阶段 3。作者可要求修改，修改后重新确认。

### 5.3 阶段 3 的章节树与梯度规划

阶段 3 产出落到 `00-教材设计.md`，包含：

- **章节树**：每章标题 + 一句话定位 + 该章承载哪些持久理解（强制对齐，未对齐的章要砍或改）。
- **每章例题/习题计划**：每章列出预期的示范例题、引导练习、独立习题数量，每条标注 Bloom 层级（记忆/理解/应用/分析/评价/创造）。
- **全书 Bloom 梯度检查**：主 skill 汇总各章标注，输出梯度报告。若发现某层级占比异常（如全书例题 ≥60% 堆在"理解"层，或完全无"应用/分析"层），必须告警并建议在哪些章补题，作者确认后进阶段 4。
- **表现性任务**：1–3 个综合性任务，对应迁移目标。

**Gate 规则**：阶段 3 是次要 gate——作者确认章节树和梯度报告后放行。

### 5.4 阶段 4 的章节内部固定节奏（write-textbook-chapter）

每章正文按四段式模板生成（借鉴 mcpmarket 模块化思路 + 脚手架/认知负荷理论）：

```
1. 概念讲解        ——引入新概念，建立在前一章之上，不跳跃
2. 示范例题        ——完整解题过程（worked example，脚手架），由 generate-textbook-exercises 生成并验证
3. 引导练习        ——半独立练习，给部分提示或脚手架
4. 独立习题        ——无提示变式 + 少量开放题，附参考答案或解题路径
```

**输入契约（单章写作 skill）**：该章大纲切片 + UbD 五件套（大概念/持久理解/核心问题）+ 已完成章节的术语表。不传入其他章正文。

**输出契约**：该章 Markdown 正文 + 该章新增术语（追加到术语表）+ 该章例题/习题的 Bloom 标注回写。

### 5.5 阶段 5 审核定稿

主 skill 自检清单（非 gate，但必须全跑一遍并报告）：

- [ ] 每章是否都承载了至少一条持久理解（章节对齐走查）
- [ ] 全书 Bloom 梯度是否实际达标（正文写完后复核，不只看计划）
- [ ] 术语全书一致（术语表 vs 各章用词）
- [ ] 例题答案全部已验证（无"未验证"残留，除非明确标注来源）
- [ ] 章节间循序渐进，无前置知识跳跃

任何一项不通过，报告具体位置和建议，由作者决定修不修。

### 5.6 真实性红线（贯穿全程）

复用 tutorial-writer 的真实性原则，对 STEM 教材强化为：

- **例题答案必须真算一遍**：generate-textbook-exercises 对每道示范例题执行计算验证（逐步推导并核对最终答案）。算不对的不能输出。
- **习题附参考答案或解题路径**：独立习题至少给出关键步骤或最终答案，不出现"留给读者"的悬空。
- **不能验证的内容明确标注**：若某题涉及开放讨论或数值实验，标注"需作者确认"而非编造结论。

## 6. 非功能需求

| 维度 | 要求 |
|------|------|
| **中断可续** | 主 skill 靠 `.progress.json` + 落盘文件，任意阶段中断后重入能定位到上次进度，不丢上下文。 |
| **上下文效率** | 单章写作 skill 输入只含必要切片，单次处理 ≤ 单章规模，支持 10+ 章。 |
| **可复用** | write-textbook-chapter 和 generate-textbook-exercises 可被教材项目外的其他场景独立调用。 |
| **可观测** | 每个阶段跑完打印一行进度；每个 gate 显式停顿并说明在等什么。 |
| **可维护** | references 与 SKILL.md 分离；UbD/Bloom/章节模板等可独立更新。 |
| **语言** | 教材正文默认中文；UbD/Bloom 等术语在中英对照后统一用一种。 |

## 8. 里程碑与验收标准

| 里程碑 | 内容 | 验收标准 |
|--------|------|----------|
| **M1｜骨架搭建** | 四个 skill 目录 + SKILL.md 骨架 + references 全部新建完成 | 各 SKILL.md 能被正确触发；references 内容完整无占位 |
| **M2｜大纲 skill 跑通** | design-textbook-outline 完成阶段 1-3，含两个 gate | 用一门真实 STEM 学科（如"线性代数入门"）跑通，产出 `00-教材设计.md`，五件套齐全，Bloom 梯度报告正常告警 |
| **M3｜单章 skill 跑通** | write-textbook-chapter + generate-textbook-exercises 跑通 | 从 M2 的章节树挑一章写出正文，四段式完整，例题答案经验证正确 |
| **M4｜长教材端到端** | 主 skill 调度写完一本 10+ 章教材 | 端到端跑通，中途人为中断一次能续写，最终自检清单全过 |
| **M5｜验收** | 用第二门学科（如"数据结构基础"）复跑一遍 | 无需改 skill 即可产出合格教材，证明不是过拟合到第一门学科 |

**v1 完成定义（DoD）**：M1–M5 全过，且至少两本不同 STEM 学科教材产出通过自检清单。

## 9. 风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| **拆分增加工程复杂度** | 调度逻辑、状态管理出错概率上升 | 主 skill 的 `.progress.json` 状态机要简单明确；M2–M3 先验证单个子 skill，M4 才组装 |
| **UbD gate 门槛偏高** | 作者得多花一轮想"持久理解"，可能嫌麻烦绕过 | gate 是硬性质量底线，不省；但可在 references 里给足示例降低思考成本 |
| **Bloom 梯度检查流于形式** | 标了层级但不真正指导出题 | 强制全书梯度报告 + 阶段 5 复核正文（不只看计划）；异常必须告警 |
| **STEM 例题计算验证不可靠** | 答案仍可能错 | generate-textbook-exercises 对计算类题逐步推导核对；数值题尽量给出可复算的中间步骤 |
