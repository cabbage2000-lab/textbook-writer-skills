# UbD 逆向设计框架（教材版落地指南）

本文档是 design-textbook-outline skill 阶段 2（UbD 预期结果设计）的内核依据，源自 Wiggins & McTighe《Understanding by Design（追求理解的教学设计）》，已针对"写 STEM 概念型教材"场景做落地化改写。

## 1. 为什么逆向设计

UbD 的核心主张是**以终为始**——先想清楚学生该带走什么，再设计怎么证明他们学会了，最后才动手写内容。三阶段顺序不可颠倒：

| 阶段 | 名称 | 回答的问题 | 在本 skill 组合中的位置 |
| ------ | ------ | ----------- | ------------------------ |
| 一 | 确定预期结果 | 学生学完该理解什么、能迁移什么？ | design-textbook-outline 阶段 2（核心 gate） |
| 二 | 确定评估证据 | 拿什么证明学生真的学会了？ | design-textbook-outline 阶段 3（例题/习题计划 + 表现性任务） |
| 三 | 设计学习体验 | 具体怎么教、怎么写？ | write-textbook-chapter 逐章落实 |

教材最常见的病是**知识点堆砌无主线**：一章章罗列概念，读者读完只记得零散名词。病根在于作者跳过了阶段一，直接从"我知道哪些内容"出发组织章节。UbD 阶段一就是解药——先逼出五件套，后面每一章、每道题都必须能追溯到它。

## 2. 五件套逐项落地

阶段 2 的产出固定为五件套，缺一不可。以下每项给出定义、质量标准、好例与坏例（示例统一用"线性代数入门"教材）。

### 2.1 大概念（Big Ideas）

**定义**：学科中有解释力和组织力的核心概念主张，能把一批零散知识点串成整体。

**数量约束**：3–7 个。少于 3 个说明教材范围可能太窄或提炼不足；多于 7 个说明还没真正提炼——大概念要"少而精"。

**质量标准**：

- 是一个"主张"或"视角"，不是一个名词术语；
- 能串起 ≥3 个零散知识点；
- 在全书多章反复出现并加深。

**好例**："线性变换——矩阵不是数字表格，而是空间变换的代数表示。"（它串起了矩阵乘法、行列式、特征值、基变换等一批知识点，并给出统一视角。）

**坏例**："矩阵。"（这只是名词，没有主张，什么也解释不了；"第二章讲矩阵"式的目录思维正是要避免的。）

### 2.2 持久理解（Enduring Understandings）

**定义**：希望学生在课程结束、忘掉细节之后仍能保留并应用的核心洞见——"忘掉细节后剩下的东西"。

**句式强制**：每条必须写成"**学生将理解：……**"。这个句式逼你写出一个完整的、可以被理解的命题，而不是一个话题标签。

**数量建议**：4–8 条（与大概念数量相称，每个大概念至少一条展开）。

**质量标准**：

- 是洞见（insight），不是知识点复述；
- 检验法：把句子里的"学生将理解"删掉，剩下的部分如果是一条定义或公式，就不合格；如果是一个需要整章铺垫才能体会的判断，就合格。

**好例**："学生将理解：解线性方程组的过程本质上是在问'目标向量是否落在系数矩阵的列空间里'。"

**坏例**："学生将理解矩阵乘法的定义。"（这是记忆层的知识点，不是洞见——学生背下定义不等于获得了任何持久的东西。）

### 2.3 核心问题（Essential Questions）

**定义**：指向大概念的开放性问题，能引发持续思考、在全书多章反复回来，而不是有标准答案的习题。

**数量约束**：2–5 个。

**质量标准**：

- 无标准答案，或答案随理解深入而不断丰富；
- 能贯穿多章：第 2 章第一次遇到它，第 6 章能给出更深的回答；
- 学生读完全书后应该能对它做出有质量的回应。

**好例**："为什么同一个变换在不同基下长得完全不一样？我们该如何选择'好'的基？"（第一次讲基时提出，讲到特征值、对角化时再回来，层层加深。）

**坏例**："矩阵乘法怎么算？"（有标准答案，做一遍就关闭了，无法贯穿全书——这是习题不是核心问题。）

### 2.4 迁移目标（Transfer Goals）

**定义**：学生能把所学应用到**新情境**的能力——这是教育的终极目标。

**句式建议**："学生能在<新情境>中运用<所学>做<事情>。"

**数量建议**：2–4 条。

**质量标准**：

- 情境必须是教材没有直接教过的（否则只是应用不是迁移）；
- 必须在阶段 3 被至少一个表现性任务覆盖，否则就是空话。

**好例**："学生能在遇到任何降维、压缩、推荐系统场景时，主动识别其中的低秩结构并用矩阵分解语言描述它。"

**坏例**："学生能解教材上的习题。"（习题是教过的情境，这是练习不是迁移。）

### 2.5 学习目标（Learning Objectives）

**定义**：可验证的、行为化的目标条目，说明学生"能做什么"。

**动词强制**：每条必须使用 Bloom 对应层级的动词（动词表见同目录 [bloom-levels.md](bloom-levels.md) 第 2 节），并在句末标注层级。**禁用**"理解、掌握、了解、熟悉"等不可验证的模糊动词。

**数量建议**：全书 8–15 条（约每章 1–2 条落点），覆盖 ≥4 个 Bloom 层级。

**质量标准**：

- 每条可验证——能想象出一道题或一个任务来检验它；
- 每条能追溯到某条持久理解（见第 3 节一致性检查）；
- 覆盖多个 Bloom 层级，不全堆在一层。

**好例**："能用高斯消元法求解 3×4 增广矩阵对应的方程组，并判断解的个数（Bloom：应用）。"

**坏例**："理解高斯消元法。"（"理解"无法验证——学生怎样算"理解了"？说不清就无法出题、无法评估。）

## 3. 五件套之间的一致性检查

五件套不是五个独立清单，它们之间有强制的追溯关系。阶段 2 呈现给作者前，先自查：

| 检查项 | 规则 | 不通过的处理 |
| -------- | ------ | ------------- |
| 学习目标 → 持久理解 | 每条学习目标须能追溯到至少一条持久理解 | 追溯不到的目标要删除，或补写它服务的持久理解 |
| 核心问题 → 大概念 | 每个核心问题须指向至少一个大概念 | 指不到的问题是无主线的"趣味题"，删除或改写 |
| 迁移目标 → 表现性任务 | 每个迁移目标须在阶段 3 被至少一个表现性任务覆盖 | 阶段 3 校验；未覆盖则补任务或明示放弃该目标 |
| 持久理解 → 大概念 | 每条持久理解应是某个大概念在具体方向上的展开 | 孤立的持久理解说明大概念清单不完整 |

## 4. 五件套在后续阶段如何使用

五件套经作者确认后（合称 **UbD 五件套**）即成为全流程的对齐基准：

- **阶段 3（章节设计）**：章节树中每一章必须声明它承载哪几条持久理解。承载不了任何一条的章要砍掉或改造；反过来，没有任何章承载的持久理解，要么补章，要么作者明示放弃。这条双向检查是防堆砌的硬机制。
- **阶段 4（单章写作）**：write-textbook-chapter 的输入契约携带五件套的 `{大概念[], 持久理解[], 核心问题[]}` 三项。章引言应呼应一条核心问题；概念讲解的展开方向由该章承载的持久理解决定；"本章小结"须回应本章推进了哪条持久理解。
- **阶段 5（审核定稿）**：主 skill 全书走查每条持久理解是否有章实际承载（不只看计划，看写出来的小结），并复核学习目标的 Bloom 分布是否落实。

## 5. 快速自查清单（gate 呈现前必过）

把五件套呈现给作者确认之前，逐项自问：

1. 大概念是否 3–7 个，且每个都是"主张"而非名词？
2. 每条持久理解删掉句式前缀后，剩下的是洞见还是定义复述？
3. 核心问题是否 2–5 个，且每个都能在多章反复回来？
4. 迁移目标的情境是否真的"教材没直接教过"？
5. 每条学习目标是否都用了 Bloom 动词并标注层级、可出题检验？
6. 一致性检查（第 3 节）中阶段 2 可检的三项（学习目标→持久理解、核心问题→大概念、持久理解→大概念）是否全部通过？（迁移目标→表现性任务一项留待阶段 3 校验）

任何一项不过，先修再呈现——gate 是给作者确认方向的，不是让作者替 skill 做质检的。
